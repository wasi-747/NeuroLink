📂 1. Project Overview & Architecture

NeuroVerse (developed under the project name NeuroLink) is a robust, full-stack student wellness ecosystem designed to mitigate the mental health challenges faced by university students. The platform delivers continuous mood tracking, rich-text journaling, habit reminders, stress self-assessments, anonymous peer support forums, therapist appointment booking, and targeted learning resources.

The core innovation of the platform lies in its Hybrid API-DSA Architecture, which cleanly separates high-performance, transactional web application components from data structures and algorithms (DSA) and machine learning (ML) components. This separation ensures that CPU-intensive model training, natural language inference, and graph-based recommendation traversal do not block the primary web requests.

Architectural Breakdown

  • MERN Transactional Web Backend: Built with Node.js, Express, and MongoDB. It handles user authentication, CRUD operations for profiles, journal storage, forums, therapist bookings, and administrative tasks.
  • FastAPI Machine Learning Microservice: Built with Python. It lazy-loads scikit-learn models, vectorizers, and heavy neural network libraries (such as SentenceTransformers and NetworkX). This microservice operates in isolation and communicates with the Node.js backend using a structured JSON REST API interface.
  • The Hybrid API-DSA Concept: This design integrates standard web APIs with a graph-based representation of resources. Machine learning outputs (like extracted entities or text vectors) are used to construct and traverse dynamic data structures. For example, a Knowledge Graph is constructed using cosine similarities of content embeddings, and path-finding algorithms (DFS) generate personalized study and wellness paths.

📝 Mismatch Resolution between `Combined Data` and `mental_health`

A critical hurdle in building the Sentiment Classifier was the structural discrepancy between the two text datasets. Combined Data.csv contained 53,043 rows with a column named statement and clinical classification labels in the status column (representing specific disorders). Conversely, mental_health.csv contained 27,977 rows with a column named text and binary labels in the label column representing general mental health versus crisis status.

Resolution Method:

  1. Standardizing Columns: The statement column in `Combined Data.csv` was renamed to text to match the schema.
  2. Unified Label Mapping:
    • The clinical labels in the status column were grouped: "Normal" was mapped to "NEUTRAL"; "Anxiety", "Depression", "Stress", "Bipolar", and "Personality disorder" were mapped to "NEGATIVE"; and "Suicidal" was mapped to "CRISIS".
    • The binary labels in `mental_health.csv` were mapped: 0 became "NEUTRAL" and 1 became "CRISIS".
  3. Concatenation & Pruning: The cleaned columns were extracted and concatenated, creating a single balanced dataset of 81,020 rows. Blank rows, null strings, and unrecognized symbols were dropped, followed by regex-based cleaning to strip numbers and punctuation.

⚙️ 3. Model Training Pipeline & Features

Running python training/train_all_models.py triggers the pipeline to process datasets, train models, compare performance metrics, and serialize the best-performing models to /models.

Task 1: Mood Prediction (Regression)

This task predicts a continuous numerical mood score representing the student's current baseline mood. The target variable is derived using a deterministic linear equation that represents how clinical symptoms degrade mood.

Target Feature Derivation Formulae:

$$\text{mental\_health\_score} = (\text{depression} \times 3) + (\text{anxiety} \times 2) + (\text{panic} \times 1)$$

$$\text{mood\_score} = 5 - \left(\frac{\text{mental\_health\_score}}{6} \times 4\right)$$

In this equation, depression, anxiety, and panic are input binary indicators (1 for present, 0 for absent). The weights prioritize clinical severity (depression = 3, anxiety = 2, panic = 1), mapping a fully symptomatic student to 1.0 (Very Low) and a symptom-free student to a baseline of 5.0 (Excellent).

💡 Preprocessing Rationale: StandardScaler

Standard scaling was applied to the features prior to training. The dataset combines features of vastly different scales, such as Age (ranging from 18 to 40) and binary indicator variables (0 or 1). Standard Scaling centers the data to have a mean of 0 and a standard deviation of 1. Without scaling, optimization algorithms like gradient descent or linear solvers would assign disproportionately large weights to the larger-scale age features, leading to slower convergence and fitting errors.

Evaluation & Selection: Linear Regression, Random Forest, and Gradient Boosting were evaluated. Linear Regression was selected as the production model, achieving a coefficient of determination \(R^2 = 1.000\) and Mean Absolute Error \(\text{MAE} = 0.000\).

[!NOTE] The perfect \(R^2\) of 1.000 occurs because the target variable mood_score is a deterministic, linear combination of the input features. Linear Regression mathematically discovers these exact coefficients, resulting in zero prediction error.

Task 2: Sentiment Classifier (NLP / Classification)

Classifies free-text journal logs or forum entries into NEUTRAL, NEGATIVE, or CRISIS.

  • Vectorization (TF-IDF): TfidfVectorizer was initialized with max_features=5000, an ngram_range=(1,2) to capture single words and bi-grams, and English stopword filtering. TF-IDF was preferred over standard frequency counts (Bag of Words) because it penalizes commonly occurring words across all journals (e.g., "feel", "just", "today") and amplifies words that carry specific emotional weight (e.g., "isolated", "empty", "happy").
  • Train-Test Split Stratification: When splitting the combined dataset, the parameter stratify=y_text was utilized. This forces the train and test subsets to retain the exact class distributions of the parent population. Because the CRISIS class represents a small fraction of the total samples, a standard random split risks under-representing crisis instances in the training or testing splits, leading to poor model performance and unreliable evaluation metrics.

Evaluation & Selection: Logistic Regression, Support Vector Machine (SVM), and Random Forest Classifiers were evaluated. Logistic Regression was selected (Accuracy: 0.818).

Task 3: Sleep Insights (Rule-based & Pearson Correlations)

This module reads the sleep lifestyle dataset to calculate Pearson correlation coefficients between numeric values (such as Sleep Duration, Quality of Sleep, and Stress Levels), saving them to sleep_correlations.json. It also defines thresholds in sleep_insights.json for rules to generate actionable habits and mindfulness suggestions:

  • Low Sleep: Sleep Duration < 6.0 hours (triggers sleep hygiene guides and insomnia tips).
  • High Stress: Stress Level > 7 (triggers box breathing exercises and stress-reduction courses).
  • Poor Quality: Quality of Sleep < 5 (triggers mindfulness meditation).

Task 4: FAQ Knowledge Base (Keyword Search Index)

Strips stopwords from the FAQ questions, tokenizes them, and filters out short words to build a reverse keyword-to-answer lookup map. The lookup table is saved as faq_knowledge_base.json for search retrieval.

⚠️ Troubleshooting: `/api/ml/learning/build-graph` 422 JSON Decode Error

Incident: During integration testing between the MERN web server and the FastAPI microservice, calling the /api/ml/learning/build-graph endpoint resulted in a 422 Unprocessable Content error. The console logged a JSON Decode parsing failure.

Root Cause: The microservice router uses a Pydantic schema (BuildGraphRequest) that models courses, articles, and resources as lists of ContentItem, where id, type, title, and description are required string properties. The request payload sent from the Node backend was truncated midway through the articles array, resulting in a missing description property on the last item and incomplete closing brackets (e.g. ], }). This syntax error caused Pydantic's JSON parser to crash before the router code was reached.

Resolution: The MERN backend mapping controllers (server/controllers/learning.js) were updated to ensure all MongoDB queries correctly select the description field (e.g. .select("id title description")) and map empty descriptions to empty strings if missing. The request payload assembly was verified to prevent data truncation.

🎓 5. Teacher Presentation Q&A Cheat Sheet

Use this cheat sheet to answer questions regarding model selections, parameter choices, and evaluation metrics during the presentation:

Why did you scale your input features for the Mood Prediction model?
Algorithms like Linear Regression fit parameters by calculating distances or gradients. In our dataset, input columns represent different units and scales: Age has a range of approximately 18 to 40, while clinical indicators like has_depression are binary (0 or 1). Without scaling, the algorithm would assign a disproportionately large weight to age due to its numerical range. Applying a StandardScaler normalizes each feature to have a mean of 0 and variance of 1, ensuring every input feature contributes equitably to prediction learning.
Why did you choose TF-IDF Vectorization over a simple Bag-of-Words (CountVectorizer) representation?
A simple CountVectorizer tracks word frequency counts, meaning common, neutral filler words (e.g. "really", "feel", "just", "the", "and") dominate the feature vectors without carrying emotional context. TF-IDF (Term Frequency-Inverse Document Frequency) scales down the weights of words that appear frequently across the entire corpus, while boosting the weights of rare words that carry specific emotional sentiment (e.g. "worthless", "joyful", "hopeless"). This allows our classification models to better distinguish nuances in student journals.
What is the benefit of mapping clinical statuses to three unified sentiment classes (NEUTRAL, NEGATIVE, CRISIS)?
The primary objective of the machine learning classifier in a student wellness app is triage and safety. Standard datasets contain specific clinical diagnoses (such as Bipolar, Personality Disorder, or Depression) that are difficult to classify accurately from single text entries. By mapping these categories to NEUTRAL, NEGATIVE (moderate stress needing self-care exercises), and CRISIS (distress requiring helpline links), we increase model accuracy while aligning outputs directly with UI actions (e.g. showing a standard journal response, suggesting a breathing exercise, or triggering a crisis hotline popup).
Why did you use stratification (`stratify=y_text`) when executing the train-test split for the Sentiment Classifier?
Our dataset exhibits class imbalance: NEUTRAL and NEGATIVE samples are common, whereas CRISIS samples represent a smaller fraction of the data. In a standard random split, there is a risk that the test set could contain very few or zero crisis samples, leading to unreliable metrics. Stratification ensures that the train and test splits maintain the exact same class proportions as the parent dataset, giving us a representative evaluation.
Explain the difference between R² and MAE. Why did you use both to evaluate your Mood Regressor?
R² (Coefficient of Determination) is a standardized metric indicating the proportion of variance in the target variable that is predictable from the input features (ranging from 0.0 to 1.0). MAE (Mean Absolute Error) measures the average magnitude of prediction errors in the target units (e.g., "our predicted mood score is off by 0.12 points on average"). R² tells us how well our model fits relative to a baseline mean predictor, while MAE provides a direct, interpretable measure of prediction error.
Why did your Mood Prediction Regressors achieve a perfect R² of 1.000? Is it overfitting?
No, it is not overfitting. A perfect \(R^2 = 1.000\) and \(\text{MAE} = 0.000\) is expected here because the target variable mood_score was created in preprocessing as a deterministic, linear equation from the features has_depression, has_anxiety, and has_panic. Since these three indicator features are directly included in the feature matrix \(X\), a Linear Regressor mathematically reproduces the target equation exactly. This demonstrates that the regressor is functioning correctly.
Explain how the "DSA" part of your Hybrid API-DSA Architecture works in content recommendation.
The Data Structures and Algorithms (DSA) component is implemented in learning.py:
  1. Graph Data Structure: We construct a Knowledge Graph using networkx, where each node represents a learning item (course, article, or resource) and each edge weight represents semantic similarity.
  2. Cosine Similarity: Semantic similarity is calculated between content title/description embeddings generated by a SentenceTransformer model. Edges are added if cosine similarity exceeds 0.5.
  3. Depth First Search (DFS) Traversal: When a user provides interests, we embed the search text, find the top 3 closest starting content nodes, and execute a Depth First Search (DFS) preorder traversal to build a sequential learning path. This uses fundamental graph structures and traversal algorithms in production.
